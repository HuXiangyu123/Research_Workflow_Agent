# Agent 展示与宣传图谱设计建议

日期：2026-04-17

## 1. 结论先行

如果是介绍一个真正可运行的 agent 项目，而不是只展示一个“会自动写报告”的 Demo，通常不能只画 1 张流程图。

结合业内常见做法与本项目实际结构，建议按三档来准备：

| 套餐 | 建议图数 | 适用场景 | 结论 |
| --- | --- | --- | --- |
| 最小版 | 4 张 | 面试讲解、技术汇报、一次性演示 | 足够讲清“这不是普通 RAG，而是有编排、有分工、有反馈的 agent 系统” |
| 标准版 | 6 张 | 官网宣传、项目答辩、对外介绍、投资人/主管汇报 | 最推荐，信息密度和理解成本平衡最好 |
| 完整版 | 7-8 张 | 技术白皮书、架构评审、深入售前、开源项目首页长文 | 能把运行机制、质量控制、观测性、部署边界都讲透 |

对当前仓库，`6 张`是最合适的标准答案。

原因不是这个项目“节点很多”，而是它天然包含了 6 个不同层面的信息：

1. 系统边界与产品卖点
2. LangGraph 工作流
3. multi-agent 分工与 supervisor
4. 运行时通信与 SSE 反馈
5. workspace 产物沉淀与可追溯性
6. 部署与外部依赖

## 2. 为什么这个项目适合按 agent 系统来展示

这个项目不是单纯的“用户输入一个主题，LLM 输出一篇文章”。

从代码看，它已经具备比较完整的 agent system 展示要素：

- 有正式的 LangGraph `StateGraph` 工作流，而不是脚本式串行调用。
- 有 supervisor 和专职 agent 分工，而不是单一 prompt。
- 有多种 agent 范式并存，而不是单一 ReAct。
- 有实时 SSE 过程反馈，而不是只在最后返回结果。
- 有 workspace 产物沉淀，可回看中间结果和 revision。
- 有 review / claim verification / grounded writing 约束，说明系统关注质量而不只是生成。

这意味着它适合被包装成一个“研究型 multi-agent report system”，而不是一个“报告生成脚本”。

## 3. 外部参考给出的通用展示方法

我补了一轮 web 调研，结论和本项目非常契合。

### 3.1 C4 Model 的启发

C4 官方将图分成静态结构层级，如 `System Context`、`Container`、`Component`、`Code`。同时也明确强调，不需要每一级都画，`System Context + Container` 对多数团队已经足够；`Dynamic Diagram` 应该只用于解释关键运行模式，而不是把所有细节都画进去。

这对本项目的含义是：

- 不能只画 agent workflow flowchart。
- 至少还要补一张系统边界/容器图。
- 动态交互图要挑“任务执行 + SSE 推送 + workspace 落盘”这个关键链路来画。

### 3.2 OpenAI 对 agent 的表达方式

OpenAI 在 agent 实践指南里把 agent 基础拆成 `Model + Tools + Instructions`，并强调 manager pattern：中心 agent 负责统一编排，按需把任务分发给专门 agent，再把结果整合回来；同时强调 guardrails / human intervention / quality control。

这对本项目的含义是：

- 宣传图不能只讲“有哪些节点”，还要讲“谁在统一调度，谁负责检索，谁负责写作，谁负责 review”。
- 要把 review / grounding / claim verification 作为卖点，而不是隐藏在实现细节里。

### 3.3 LangChain / LangGraph 对 multi-agent 的表达方式

LangChain 官方 multi-agent 文档强调，多 agent 适合三类场景：

- 单 agent 工具太多，容易选错
- 不同任务需要专门上下文
- 工作流存在顺序约束

官方 supervisor / subagent 说明也明确指出：当你需要集中式控制、多专长 agent 协作、而子 agent 不需要直接和用户对话时，supervisor pattern 是合适的。

这正好符合当前仓库：

- planner、retriever、analyst、reviewer 是明显的专职分工
- research workflow 有明确顺序约束
- 统一入口仍然是 supervisor / graph，而不是多个 agent 各自对外暴露

## 4. 基于当前仓库，最推荐的图谱组合

### 4.1 最小版：4 张

适合面试、短讲、10 分钟内说明项目。

1. 总览架构图
2. Research workflow 流图
3. Multi-agent 分工图
4. 运行时时序图

### 4.2 标准版：6 张

适合官网、宣传页、技术答辩。对这个项目最推荐。

1. `Hero Architecture Diagram`
2. `Research StateGraph / Workflow Diagram`
3. `Supervisor + Specialized Agents Diagram`
4. `Runtime Sequence / Communication Diagram`
5. `Workspace Artifact Lineage Diagram`

适合白皮书或深度架构讲解。

## 5. 这 6 张图分别应该画什么

下面不是抽象建议，而是直接对应到当前项目真实代码。

### 图 1：Hero Architecture Diagram

这是最重要的一张图，负责“第一眼让人知道这个项目厉害在哪”。

**回答的问题**

- 这是个什么系统？
- 与普通单次 LLM 调用有什么区别？
- 对外能讲什么卖点？

**建议画面**

从左到右分 5 层：

1. User / Researcher
2. Frontend UI
3. API + Task Runtime
4. LangGraph + Supervisor + Agents
5. External Tools / Persistence

**本项目应出现的元素**

- Frontend：GraphView、Thinking Panel、Report Preview、Workspace Inspector
- Backend：FastAPI `/tasks`、`/tasks/{id}/events`、background task execution
- Orchestration：LangGraph `StateGraph` + supervisor
- Specialized agents：Planner / Retriever / Analyst / Reviewer
- Tools & systems：SearXNG、arXiv / DeepXiv、LLM provider、PostgreSQL、`output/workspaces`

**最应该强调的宣传话术**

- From topic to grounded survey, with visible reasoning, artifact persistence, and review gating.

**代码依据**

- `src/research/graph/builder.py`
- `src/research/agents/supervisor.py`
- `src/api/routes/tasks.py`
- `src/api/routes/workspaces.py`
- `frontend/src/App.tsx`

### 图 2：Research StateGraph / Workflow Diagram

这是“流程图”，但不要把它当成全部。

**回答的问题**

- 系统怎么一步步完成 research survey？
- 哪些节点是固定阶段，哪些有条件跳转？

**建议画面**

按真实 graph 画：

`clarify -> search_plan -> search -> extract -> extract_compression -> draft -> review -> persist_artifacts`

并标出两个关键条件路由：

- `clarify` 后如果 `awaiting_followup` 或没有 `brief` 则结束
- `review` 只有 `review_passed=true` 才进入 `persist_artifacts`

**为什么这张图重要**

- 它证明项目是 graph orchestration，不是 prompt 链。
- 它能清楚解释“为什么 review 挡住了最终成文”。

**代码依据**

- `src/research/graph/builder.py`

### 图 3：Supervisor + Specialized Agents Diagram

这是“agent 图”，也是最容易被宣传成亮点的一张。

**回答的问题**

- multi-agent 到底体现在哪里？
- 每个 agent 负责什么？
- 为什么要分 agent，而不是一个大模型全做？

**建议画面**

中间一个 `AgentSupervisor`，向下连接四个核心 agent：

- `PlannerAgent` -> Plan-and-Execute
- `RetrieverAgent` -> TAG
- `AnalystAgent` -> Reasoning via Artifacts
- `ReviewerAgent` -> Reflexion

旁边注明：

- supervisor 使用官方 `create_supervisor`
- worker agent 使用官方 agent / graph 模式
- canonical order 仍由主 graph 保证

**宣传层面的重点**

这张图最适合讲“不是一个 agent 套所有事情，而是把 planning、retrieval、writing、review 拆成不同认知角色”。

**代码依据**

- `src/research/agents/supervisor.py`
- `src/research/agents/planner_agent.py`
- `src/research/agents/retriever_agent.py`
- `src/research/agents/analyst_agent.py`
- `src/research/agents/reviewer_agent.py`

### 图 4：Runtime Sequence / Communication Diagram

这张图是很多 agent 项目宣传里缺失的，但对“显得真实可用”非常关键。

**回答的问题**

- 用户点击开始后，系统内部发生了什么？
- 前端为什么能实时看到进度、thinking、report snapshot？

**建议画面**

参与者建议画成 7 个泳道：

- User
- Frontend
- FastAPI
- Task Runtime
- LangGraph
- Workspace
- PostgreSQL

关键事件链路：

1. Frontend 调 `POST /tasks`
2. Backend 创建 task，绑定 `workspace_id`
3. 后台线程启动 graph
4. graph 每个 node 发出 `node_start` / `node_end`
5. SSE `/tasks/{id}/events` 持续推送状态和 `thinking`
6. draft / review 阶段推送 `report_snapshot`
7. node 结果同步写入 `output/workspaces/...`
8. 任务快照与结果写入 PostgreSQL
9. Frontend 实时刷新 graph、thinking panel、report preview、workspace inspector

**为什么必须有这张图**

它能把“能运行”与“能被看见”一起讲清楚。很多项目只会说 agent 很智能，但讲不清运行态反馈，这张图能直接拉开档次。

**代码依据**

- `src/api/routes/tasks.py`
- `frontend/src/hooks/useTaskSSE.ts`
- `frontend/src/App.tsx`
- `src/db/task_persistence.py`
- `src/agent/output_workspace.py`

### 图 5：Workspace Artifact Lineage Diagram

这张图是本项目很适合对外讲的差异化亮点。

**回答的问题**

- 为什么这个系统不是黑盒？
- 报告是怎么被一步步证据化构建出来的？
- 中间结果如何落盘和追溯？

**建议画面**

按 artifact 演进顺序画：

- `brief.json`
- `search_plan.json`
- `rag_result.json`
- `paper_cards.json`
- `comparison_matrix.json` / `writing_scaffold.json` / `writing_outline.json`
- `draft.md`
- `review_feedback.json`
- `claim_verification.json`
- `revisions/*.md`
- `report.md`

并在图旁说明：

- 同一 task 落在同一 workspace 下
- 用户可以回看 task 级产物
- review 失败也会保留 revision，便于诊断

**宣传层面的重点**

这张图很适合讲“可审计 research pipeline”。

**代码依据**

- `src/agent/output_workspace.py`
- `src/api/routes/workspaces.py`

### 图 6：Container / Deployment Diagram

这张图的作用不是强调 agent，而是回答“它怎么接入真实系统”。

**回答的问题**

- 这个系统部署时有哪些容器/服务？
- 哪些是项目自身组件，哪些是外部依赖？
- 数据流与边界如何划分？

**建议画面**

建议至少包含：

- Browser
- React + Vite frontend
- FastAPI backend
- LangGraph runtime
- PostgreSQL
- SearXNG
- LLM provider
- file-based workspace storage `output/workspaces`

如果要更工程化，可以再标：

- `.env` config
- checkpointer / task persistence
- optional FAISS / pgvector

**为什么要画**

对于主管、架构师、面试官，这张图会把项目从“一个研究脚本”提升成“可部署的软件系统”。

**代码依据**

- `docs/current-architecture-and-usage.md`
- `src/db/task_persistence.py`
- `src/agent/output_workspace.py`

## 6. 如果要扩充到 7-8 张，优先加哪两张

### 6.1 Grounding / Review Quality Loop Diagram

这张图特别适合当前项目，因为最近几轮实验已经暴露出：

- 不是“没检索到材料”
- 而是 review 在卡最终成文
- claim grounding / citation density / confidence 仍然是关键矛盾

所以很值得单独画一张：

- `draft_report`
- `resolve_citations`
- `verify_claims`
- `claim_verification`
- `review_feedback`
- `revision`
- `final_report`

这张图最适合讲“为什么系统对质量敏感，而不是只追求一键出文”。

### 6.2 Frontend Observability / User Journey Diagram

这张图更偏产品宣传。

可以画成用户视角：

1. 新建 workspace
2. 发起 task
3. 观看 graph 动态执行
4. 看到 thinking 流
5. 查看 report snapshot
6. 回看 workspace 历史产物

这张图的价值在于：把“agent”从后端能力变成用户能感知的产品体验。

## 7. 这类项目最容易画错的地方

### 错法 1：只有一张流程图

问题：

- 流程图只能解释步骤，解释不了角色分工、系统边界、实时反馈、产物沉淀。

后果：

- 看起来像 prompt pipeline，不像 multi-agent system。

### 错法 2：把每个 node 都画成 agent

问题：

- 会把 graph node、tool、agent、storage、UI 概念混在一起。

对当前项目尤其要避免：

- `clarify`、`extract`、`persist_artifacts` 更适合画成 workflow node
- `PlannerAgent`、`RetrieverAgent`、`AnalystAgent`、`ReviewerAgent` 才适合画成 agent role

### 错法 3：没有 runtime 图

问题：

- 讲不清 SSE、workspace、task persistence 是怎么协同工作的。

后果：

- 前端看起来像“只是一个结果展示页”，而不是实时 agent cockpit。

### 错法 4：没有 artifact 图

问题：

- 讲不清 evidence 如何进入 draft，再进入 review 和 final report。

后果：

- 质量控制能力会被低估。

## 8. 面试 / 宣传时最推荐的出图顺序

如果只给你 5-8 分钟介绍这个项目，建议按下面顺序讲：

1. `Hero Architecture Diagram`
   先让人知道这是一个“可观察、可追溯的 research multi-agent system”。
2. `Research Workflow Diagram`
   说明主线流程是如何一步步完成 survey 的。
3. `Supervisor + Agents Diagram`
   再讲为什么这里不是单 agent，而是分工协作。
4. `Runtime Sequence Diagram`
   证明系统不是 PPT agent，而是有实时运行反馈。
5. `Workspace Artifact Diagram`
   最后强调可审计、可复盘、可诊断。

如果还有时间，再加：

6. `Container / Deployment Diagram`
7. `Grounding / Review Quality Loop Diagram`

## 9. 最终建议：这个项目到底应该准备多少种图

### 面试最小集

`4 张`

- 总览架构图
- 工作流图
- multi-agent 分工图
- 时序图

### 对外宣传标准集

`6 张`

- 总览架构图
- 工作流图
- multi-agent 分工图
- 时序图
- workspace / artifact 图
- 容器 / 部署图

### 技术白皮书完整集

`7-8 张`

- 标准集 6 张
- grounding / review 质量闭环图
- frontend observability / user journey 图

对当前仓库，我的建议非常明确：

不要停留在“流图、通信图、架构图”这三类基础图。

真正能把项目讲出竞争力的，是把它扩成下面这 6 类：

1. 系统总览图
2. 工作流状态图
3. agent 分工图
4. 运行时交互图
5. 产物流转图
6. 部署边界图

这 6 类图刚好覆盖“产品、架构、agent、运行、质量、工程化”六个维度，是当前项目最合适的展示组合。
