# PaperReader Agent — 核心技术栈

## 1. 技术栈全景

| 层次 | 当前技术 | 在项目中的职责 |
|---|---|---|
| Backend API | FastAPI + Uvicorn + Pydantic v2 | `/tasks`、`/api/v1/skills`、`/api/v1/mcp` |
| Workflow | LangGraph + `langgraph_supervisor` | report graph、research graph、official supervisor |
| LLM 接入 | OpenAI-compatible chat clients | reason、quick、chat、review、draft |
| Persistence | PostgreSQL + SQLAlchemy 2 | task snapshots、final reports |
| Workspace 输出 | filesystem under `output/workspaces/` | 中间 artifacts、live preview 文件源 |
| Retrieval | SearXNG + arXiv API + DeepXiv + FAISS | 多源召回、metadata enrich、本地证据 |
| Frontend | React 19 + TypeScript 5 + Vite 8 + Tailwind CSS 4 | task UI、graph、preview、tool log |
| Streaming | SSE + `StreamingResponse` | 节点状态、artifact ready、report snapshot |
| Skills/MCP | skills registry + skill orchestrator + MCP adapter | 本地 skill、MCP prompt/tool/resource |

## 2. 用了什么方法（Use What）

### 2.1 API 与长任务

- FastAPI 负责 HTTP 接口。
- 长任务不走同步阻塞，而是提交 task 后异步运行。

### 2.2 工作流编排

- Report workflow 和 research workflow 都用 `StateGraph`。
- 多智能体编排用 `create_supervisor` 和 `create_react_agent`。

### 2.3 持久化

- PostgreSQL 存 durable state。
- workspace 文件夹存可读工件。
- checkpointer 用 `MemorySaver` 或 `PostgresSaver`。

### 2.4 检索与证据

- SearXNG 负责广度召回。
- arXiv API 负责精度和 metadata。
- DeepXiv 提供 TLDR、趋势与补充信息。
- 本地证据再通过 chunk/索引层供后续写作和验证复用。

### 3.5 Checkpointer

```python
from langgraph.checkpoint.memory import MemorySaver

def get_langgraph_checkpointer(namespace: str = "default") -> BaseCheckpointSaver:
    backend = os.getenv("LANGGRAPH_CHECKPOINT_BACKEND", "memory").strip().lower()
    if backend in {"", "memory", "inmemory", "in_memory"}:
        key = ("memory", namespace, "")
        if key not in _CHECKPOINTERS:
            _CHECKPOINTERS[key] = MemorySaver()
        return _CHECKPOINTERS[key]
```

## 4. 技术选型为什么合理

### LangGraph

- 这个项目的核心不是聊天，而是多阶段工作流。
- 需要显式节点、条件边、事件、checkpoint。
- 因此比“单 prompt + 手写 if/else”更合适。

### PostgreSQL

- 仓库硬规则已经明确禁止 SQLite。
- task snapshot、report record、JSON payload 更适合 PostgreSQL。

### Workspace 文件输出

- 如果只靠数据库，用户很难直接看中间工件。
- 如果只靠内存，刷新页面和重启进程后信息就丢了。
- 当前是数据库和 workspace 双层配合。

### SSE

- 当前需求是单向事件流，不需要双向长连接。
- `report_snapshot` 与 `artifact_ready` 非常适合 SSE。

## 5. 面试时怎么回答“用了什么技术”

不要只背栈名，建议直接按下面口径回答：

1. API 层用 FastAPI，把长任务统一建模成 `/tasks`。
2. 编排层用 LangGraph，分别维护 report graph 和 research graph。
3. 多 agent 层用官方 `create_supervisor + create_react_agent`。
4. 持久化层用 PostgreSQL，工件层用 workspace 文件输出。
5. 检索层是 SearXNG + arXiv + DeepXiv + 本地证据层。
6. 前端用 React + SSE 做图谱与 live preview。
